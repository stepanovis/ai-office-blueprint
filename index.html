Where things live

Three distinct environments. Each has a clear purpose and a single source of truth. Nothing is duplicated across them — everything else is fetched at the moment it is needed.

On each desktop

Employee workstation

macOS / Windows · one per associate
  • Claude Cowork app — the AI surface the associate works in.
  • MCP plugin (auto-installed) — exposes the central store to Cowork.
  • Local connectors — to personal Gmail, Calendar, Drive.
  • Bearer token — in the OS keychain, never in a prompt.
Central

Central store (server)

A small VPS · Fly.io / Railway / AWS
  • API — the one way to read or write anything.
  • Postgres + pgvector — metadata, versions, embeddings.
  • S3 bucket — raw original files.
  • Inbox table — single entry point for all ingested events.
  • Row-Level Security — tenant isolation at the DB level.

  • Version-controlled — every change reviewed via pull request.
The partner's browser is the fourth surface — used for admin tasks (inviting clients, promoting artifacts, rotating tokens).
03

What Cowork does, and what it does not

Cowork is the interface, not the store. Every associate opens Cowork as their daily workspace. Inside it, they drive the AI through natural instructions and slash commands. The AI reads local files, browses the central store, composes deliverables, sends drafts for review.

Cowork does not store anything long-term. Chats and in-flight context are local and ephemeral. Any artifact worth keeping is pushed to the central store through the plugin — otherwise it effectively disappears with the session.

Every associate has the same Cowork. Same installed plugins, same skills, same connector permissions, same rulebook. Different local files, different device. The uniformity is what makes the system predictable.

Cowork's job, in one list
  • 1.Run multi-step instructions across local files and installed apps.
  • 2.Call tools exposed by the plugin — read and write the central store.
  • 3.Follow the centrally shipped skills and rules without local management.
  • 4.Collect from personal channels (email, drive, calendar) and forward the results.
  • 5.Never become the system of record — that role belongs to the central store.

Associates do not edit skills or rules on their own machines. They open a pull request in the GitHub repo, the partner reviews and merges, and the marketplace redistributes the new version.
05

What a document actually is

Every document in the system is a container with exactly two representations: the original file, and a canonical Markdown version of its contents. Nothing more.

The AI operates exclusively on the canonical Markdown. The client sees the raw original, rendered on the fly by a viewer component chosen for the file's mime type — there is no pre-generated preview in the store.

When the raw is audio or video, its canonical is a timestamped transcript — still Markdown, produced by Whisper during extraction. So transcript is not a third format; it is simply what canonical looks like for that kind of source.

Artifact
id · client_id · type · tags · visibility
v3

Original file
pdf · docx · mp3 · eml · any mime type — the source of truth, stored in S3.
Canonical
Normalized Markdown
Uniform format across all sources. For audio/video this is a timestamped transcript. The AI reads only this.
06

How documents get ingested

Ingest has two parallel tracks that merge into one final step. When a source has a native Cowork connector, Cowork itself performs the collection, extraction, and triage — it is the ingester. When a source has no connector (Telegram, Fireflies, third-party forwards), a small integrator service collects instead, a deterministic extractor converts the raw file, and a Cowork agent handles triage from that point on. In both tracks, Cowork is the active worker whenever judgement is required.

Cards highlighted in the warm color are performed by Cowork. The deterministic extractor in Track B exists only because the integrator has delivered a raw binary without the rich context Cowork has when it reads its own connectors.
Stage 1 · Collect
Bring raw material in

Track A: Cowork reads the associate's own channels (Gmail, Drive, Calendar) through its native connectors. It already knows whose mailbox it is reading — context travels with the data.

Track B: A small server-side integrator listens to an external source (Telegram user client, Fireflies webhook, email alias, Slack webhook) and drops a raw file plus minimal metadata into the inbox_events table. Deduplication by source id, full audit trail.

Stage 2 · Extract
Raw → canonical Markdown

Track A: Cowork does this inline — it is already reading the content, it just needs to commit the canonical MD to storage.

Track B: A deterministic worker converts the raw binary: pdf → pdfplumber, docx → pandoc, eml → email-parser, mp3/mp4 → Whisper, html → readability. No AI decisions here — just format conversion.

Stage 3 · Cowork as triage
Classify & publish

In both tracks, a Cowork agent reads the canonical MD and decides:

  • · Which client_id this belongs to.
  • · What type it is (contract, email, meeting, note, spec, invoice).
  • · What tags to apply.
  • · Default visibility = internal, with promotion rules from AI_Rules.md.
  • · Unclear cases → buffer, where a human decides in the admin panel.

In Track A this is the same agent that collected and extracted. In Track B it is a long-running Cowork process on the server. Same MCP plugin, same skills, same rulebook.

Stage 4 · Store + Embed
Commit the container

One transaction writes the artifact, its version, both representations (raw + canonical), and the embedding chunks into Postgres + pgvector.

From this point the artifact is fully available to associates, and — once promoted — to the client.

Collection paths — which track each uses

Four ways content enters the system. The first two are Cowork-driven (Track A); the last two are external (Track B).

Path 01 · Employee push · Track A

Associate runs /save-artifact in Cowork
· Cowork itself performs collect + extract + triage in one step
· MCP plugin attaches the bearer token
· Client_id is already set by the associate's project context
Used for: anything the associate produces directly in Cowork.

Path 02 · Cowork connector · Track A

Gmail / Drive / Calendar forward
· Cowork's native connector reads the associate's account
· A scheduled or on-demand skill reads, extracts, and forwards in one go
· Cowork triages inline using what it already knows about the sender
Used for: email threads, shared drive files, calendar events.

Path 03 · Integrator service · Track B

Telegram · Fireflies · Slack · etc.
· A small server-side service listens via webhook or user client
· Uploads the raw file, writes the inbox event
· Deterministic extractor and server-side Cowork triage follow
Used for: sources without a native Cowork connector.

Path 04 · Email alias · Track B

inbox+acme@firm.tld
· Anyone (even third-party tools) can forward to a per-client alias
· The alias is auto-tagged with the client, which narrows triage
Used for: Zoom / Otter summaries, newsletters, signed contracts.
Why triage is a Cowork agent, not hard-coded rules

Classifying incoming documents is a judgement task: "is this message about ACME's Q3 filing or about their unrelated subsidiary?", "is this draft or a signed version?", "is this a compliance-sensitive item that should be frozen as internal until a partner reviews?". No static rule table will cover these.

Putting Cowork in that role means the same AI that the team uses to do the work is the one organizing the inputs — and it evolves through the same skills and rules repository (AI_Rules.md, triage skills in the plugin) that governs everything else.

How an associate pulls data from the central store

The associate asks Cowork a question. The plugin translates that into one or more tool calls to the central API. The API uses the token's scope to check which clients the associate can see, runs the query under Row-Level Security, and returns the result. Claude weaves the answer back into the conversation.

Associate
Asks in Cowork: "Summarize our last three contracts with ACME."
Cowork
Invokes plugin tool search→MCP server on the same machine
Token is fetched from the OS keychain and attached to the HTTP request.
MCP plugin
HTTP POST /search→Central API
Authorization: Bearer <token>
API
Decodes token, derives the scope of client_ids the associate can read.→Postgres query
Row-Level Security restricts results to those client_ids at the SQL level.
Postgres
Returns matching artifacts — titles, versions, canonical Markdown references, storage keys.
API
Responds to the plugin with a structured JSON list of hits.
Cowork
If deeper content is needed, calls fetch_artifact to pull the canonical Markdown of specific versions. Claude reads them and writes the summary back to the associate.
Writes follow the same path in reverse: Cowork calls /save-artifact, the API validates scope, writes the container, and enqueues extract-and-embed work.
08

How a client interacts with the store

The client never uses Cowork. They open a browser, authenticate with a magic link (or SSO for larger clients), and land on their own scoped portal. Two entry points: a document view and a RAG-backed chat.

Mode A · Documents
Browse and download

A list of artifacts marked public for this client. Filter by project, tag, or date; view inline through a viewer chosen by the file's mime type; download originals via time-limited URLs; comment without leaving the page.

  • · Loads GET /artifacts — returns only this client's rows.
  • · Viewer is picked per mime type (PDF.js for PDFs, audio player for mp3, etc.).
  • · Raw originals are served from S3 via presigned URLs.
Mode B · Chat
Ask questions across the archive

The client types a question. The answer is composed from their own documents and cited back to them.

  • · Question is embedded, then matched against this client's chunks in pgvector.
  • · Retrieved chunks go to Claude with a strict system prompt.
  • · Response streams back with clickable citations.
What makes this safe
JWT carries client_id
The browser's session token explicitly identifies which client is logged in.
SQL filter is server-side
Every query adds WHERE client_id = jwt.client_id. The request body cannot override it.
Visibility gate
Only artifacts flagged public are returned. Internal drafts and notes never leave the team.

Four surfaces, one source of truth.

The associates live in Cowork. The partner lives in the admin panel. The clients live in the portal. All three are clients of one API over one central store.
