Architecture overview

Multi-client AI office with a shared artifact store and a client-facing portal.

An architecture for a service organization that uses Claude Cowork on its employees' desktops, ingests work products from many channels, normalizes them into a single container format, and exposes them to each client through a secure web portal with a RAG-powered chat.

Claude Cowork Central API Postgres + pgvector S3 object storage Next.js portal Row-Level Security
Principle 01
Collect anywhere, write in one place.

Data flows in from many sources, but every artifact is written through a single, audited pipeline.

Principle 02
Raw is the truth; Markdown is the language of AI.

Originals are preserved for legal correctness. AI works with a normalized representation that is uniform across sources.

Principle 03
Isolation is enforced at the database level.

A misconfigured query cannot leak across clients — Row-Level Security makes cross-tenant reads physically impossible.

Storage Model

An artifact is a container of representations.

Every deliverable — a contract, a meeting transcript, an email thread, a spreadsheet — is stored as a single versioned container. Inside the container live multiple representations of the same underlying content, each optimized for a different consumer.

The AI layer only ever reads the canonical Markdown. The client only ever sees raw and preview. This separation is what makes the system both legally correct and AI-native at the same time.

Data Model

The entities and how they relate.

Every piece of data in the system belongs to exactly one client. Artifacts have versions; each version carries multiple representations and an independent set of embedding chunks. Access tokens — for both employees and clients — are scoped, revocable, and auditable.

Pipeline

From many sources to one artifact store.

Data is collected either through Claude Cowork's native connectors on each employee's desktop, or through small integrator services for sources Cowork does not yet support. Both paths land in a single inbox. A triage agent then classifies each event, extracts its canonical form, and writes it to storage with embeddings ready for retrieval.

Data Flows

How each piece of data actually moves.

Four sequences covering the system's critical paths: an employee producing an artifact, an external source being ingested, a client asking a question, and the lifecycle of an artifact from inbox to public visibility.

FLOW A

Employee writes an artifact from Cowork

The employee runs a skill in Claude Cowork. The MCP plugin attaches the bearer token transparently — it never reaches the prompt. The API validates the token's scope against the client_id taken from the project's instructions, creates a new version, and writes the representations in a single transaction.

FLOW B

External source ingested into inbox

For sources without native Cowork connectors — Telegram, Fireflies, Slack — a small integrator service listens over webhook or long-poll, deduplicates by source ID, and pushes raw events into the single inbox. A triage worker classifies the event and promotes it to an artifact.

FLOW C

Client asks a question in the portal chat

The client's JWT carries their own client_id. Every downstream query is filtered by that value at the SQL level — the application cannot choose to ignore it. Retrieved chunks are passed to Claude with a strict system prompt; the answer is streamed back with inline citations linking to source documents.

Components

What each layer is, and why it exists.

Each component has a narrow, well-defined role. Boundaries between them are explicit, which means any single layer can be replaced without rewriting the rest of the system.

Layer Made of Role
Sources Gmail, Drive, Calendar, Fireflies, Telegram, Slack, and others. Where raw data originates.
Bridges Claude Cowork connectors on desktops plus small integrator services on the backend. Bring every source to the same entry point. Use Cowork where native connectors exist; write a service otherwise.
Ingest (Inbox) Postgres inbox_events table and S3 bucket. Single point of write. Deduplicates by source ID, audits every event, preserves order.
Triage A Claude-powered worker with a centralized rulebook. Decides which client an event belongs to, what type it is, and which tags to apply.
Extract & Normalize PDF parser, office-to-markdown converter, speech-to-text, email parser. Produces canonical Markdown (and a transcript if relevant) from any raw input.
Artifact Storage Postgres for metadata and versions; pgvector for embeddings; S3 for binary representations. Persists the artifact container. Raw is kept as the truth; Markdown is the working copy for AI.
Central API A typed HTTP service with endpoints for artifacts, search, and chat. The only way anything reads from or writes to storage.
Cowork Plugin An MCP server shipped as a plugin, distributed via a private marketplace. Gives employees seamless read and write access to storage from inside Claude Cowork. Tokens live in the OS keychain and never appear in prompts.
Client Portal A Next.js application with document view, RAG chat, comments, and human escalation. The only surface a client ever sees. Scoped to their own data by design.
Auth & RLS Magic-link or SSO auth plus Postgres Row-Level Security policies. Separates employees from clients, and each client from every other, at the database level.
Governance A central AI_Rules.md plus org-provisioned skills shipped to every user. Makes sure every Cowork session operates under the same safety, compliance, and brand rules.
Client Experience

A portal the client owns, not a black box.

The portal gives every client a live view of the work produced on their behalf — organized, searchable, and answerable. No shared inboxes, no email threads, no "let me get back to you tomorrow."

Behind the chat is a retrieval-augmented pipeline that operates strictly within the client's own scope. Citations link back to the exact document and page that produced every statement.

Documents
Every deliverable, one place.
Filter by project, tag, or date. Preview inline. Download with time-limited URLs. Comment without leaving the page.
RAG chat
Ask questions about your own data.
A grounded assistant scoped to the client's public artifacts. Streamed answers, citations, and a human-escalation button whenever the answer is not in the documents.
Feedback loop
Comments become work items.
A comment in the portal creates a notification back to the team's Cowork, closing the loop between client and delivery without a separate tool.
Operating Rules

Six rules that keep the system coherent.

Architecture decisions compound. These are the non-negotiable invariants — the ones that remain true whether the system has two clients or two hundred.

Rule 01
Writes are always centralized.
Reads and collection can be distributed. Storage has a single writer: the central API.
Rule 02
Raw for the client, Markdown for the AI.
Originals are evidence. Normalized Markdown is the working substrate for every AI operation.
Rule 03
Tenant isolation lives in the database.
Row-Level Security in Postgres makes cross-tenant reads impossible, regardless of application bugs.
Rule 04
Tokens never appear in prompts.
Secrets live in the OS keychain or environment. The MCP server attaches them to outbound requests — the model never sees them.
Rule 05
Default visibility is internal.
A deliverable becomes visible to the client only through an explicit promotion. Silence is the safe default.
Rule 06
There is always a path to a human.
Every AI surface — in Cowork and in the portal — exposes a first-class way to hand a question to a person.
